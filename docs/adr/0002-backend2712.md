2. Background

当前项目已经基于 pi-coding-agent 实现了基本的 Agent Loop，并提供以下能力：

* 多步模型推理与 Tool Calling
* Workspace 文件读取、写入和编辑
* Bash、Python、Node 命令执行
* Skill 加载和使用
* Artifact 生成与提交
* Conversation 与 Sandbox Workspace 绑定
* Agent Run 创建、查询、取消和 SSE 输出

但当前实现仍主要适用于短时间、单轮或弱连续性的 Agent 任务，还不具备完整的企业级 Agent Runtime 能力。

当前主要问题包括：

1. 每一轮请求都会创建新的 SessionManager.inMemory()。
2. Conversation 虽然复用了 Sandbox Session 和 Workspace，但没有真正复用 Pi SDK Agent Session。
3. 历史上下文主要通过纯文本消息重新注入，Tool Call、Tool Result、Branch 和 Compaction 等状态无法完整恢复。
4. Bash、Python 和 Node 执行采用同步等待方式，不支持长任务、后台进程和交互式进程。
5. 命令日志只有执行完成后才能返回，缺少实时 stdout/stderr。
6. Tool 执行台账尚未完整覆盖所有 Tool Path。
7. 文件附件没有以结构化上下文传递给 Agent。
8. Sandbox 虽然提供 MCP 能力，但当前 Pi Agent没有统一接入外部 MCP Tools。
9. Agent Run 执行期间不能追加 steer 或 follow-up 指令。
10. 审批通过阻塞轮询实现，无法真正暂停、释放资源并恢复。
11. 缺少 Run 级步骤数、Tool 调用数、时间、Token 和成本预算。
12. Model 配置仍以硬编码对象为主，无法根据模型真实能力动态配置。

因此，需要对当前 Agent Runtime 进行一次能力补齐，使其能够可靠支持：

* 多轮持续任务
* 长时间执行任务
* 中断与恢复
* 完整工具轨迹
* 附件处理
* 企业 MCP Tool 接入
* 用户运行中干预
* 审批暂停与继续

3. Problem Statement

当前系统中存在三个不同层次的 Session：

Conversation
├── Sandbox Session
├── Workspace
└── Pi SDK Agent Session

当前已经实现：

Conversation 1:1 Sandbox Session
Conversation 1:1 Workspace

但尚未实现：

Conversation 1:1 Logical Pi SDK Agent Session

当前实际流程为：

Conversation A
├── Turn 1 → 创建临时 Agent Session 1
├── Turn 2 → 创建临时 Agent Session 2
└── Turn 3 → 创建临时 Agent Session 3

每一轮只将部分历史文本重新注入 Agent，无法延续完整执行轨迹。

同时，Agent Tool 执行仍以同步、短命令为主：

Agent 调用 bash
→ 等待命令全部执行完成
→ 返回截断后的 stdout/stderr
→ Agent 继续执行

这种模式不适合：

* 大型项目构建
* 长时间测试
* 模型训练
* 数据处理
* 启动和调试服务
* 需要持续读取日志的任务
* 需要运行中追加指令的任务

4. Goals

本任务需要完成以下核心目标。

4.1 持久化并复用 Pi SDK Agent Session

同一个 Conversation 后续回合应恢复同一个逻辑 Pi SDK Session，而不是重新创建一个空的内存 Session。

需要完整保存和恢复：

* User Message
* Assistant Message
* Tool Call
* Tool Arguments
* Tool Result
* Tool Call ID
* Custom Entry
* Branch
* Fork
* Compaction
* Model 信息
* Thinking Level
* System Prompt 版本
* Tool/Skill/Extension 版本

目标关系：

Conversation
    1:1
Logical Pi SDK Session
    1:N
Agent Runs / Turns

⸻

4.2 支持异步和长时间进程执行

新增受管理的 Process Tool，至少包括：

process_start
process_status
process_logs
process_wait
process_write_stdin
process_signal
process_cancel

Agent 应能够完成：

启动服务
→ 查看启动日志
→ 判断服务是否就绪
→ 调用或测试服务
→ 修改代码
→ 重启服务
→ 停止服务

后台进程不能依赖 nohup 和 Shell 后台符号绕过平台管理。

⸻

4.3 实现命令日志实时流式输出

命令执行过程中应持续产生：

execution_started
stdout_delta
stderr_delta
execution_completed
execution_failed
execution_cancelled

日志需要：

* 实时提供给 Agent。
* 实时提供给前端。
* 持久化到执行日志。
* 支持按照 offset 或 sequence 分页读取。
* 支持客户端断线后续传。

⸻

4.4 完整接入 Tool Execution Ledger

所有具有副作用或需要审计的 Tool 必须经过统一执行台账。

统一状态流：

prepared
→ awaiting_approval
→ executing
→ succeeded / failed / cancelled / unknown

覆盖范围至少包括：

* read
* write
* edit
* apply_patch
* bash
* process
* Skill Tool
* MCP Tool
* Artifact Tool
* Approval Tool
* 外部 HTTP Tool

每次 Tool Call 必须持久化：

tool_call_id
run_id
session_id
conversation_id
workspace_id
tool_name
arguments
status
execution_id
idempotency_key
started_at
finished_at
result_summary
error

⸻

4.5 支持结构化附件上下文

用户上传附件后，本轮 Agent Message 必须显式包含：

attachment_id
filename
workspace_path
mime_type
size
upload_time

Agent 不应依靠扫描整个 uploads/ 目录猜测用户上传了哪个文件。

示例：

{

  "attachments": [
    {
      "attachment_id": "att_xxx",
      "filename": "report.xlsx",
      "path": "uploads/att_xxx/report.xlsx",
      "mime_type": "application/vnd.openxmlformats-officedocument.spreadsheetml.sheet"
    }
  ]
}

4.6 建立统一 Tool Registry 和 MCP 接入

当前 Agent Tool 来源需要统一管理：

ToolRegistry
├── Sandbox Tools
├── Process Tools
├── Skill Tools
├── MCP Tools
├── Artifact Tools
└── Enterprise HTTP Tools

MCP 接入至少支持：

* MCP Server 注册
* Tool Discovery
* Tool Schema 转换
* Tool Allowlist
* 用户和组织权限校验
* Approval Policy
* Tool Execution Ledger
* 超时和重试

⸻

4.7 支持运行中追加指令

需要暴露：

POST /runs/{run_id}/steer
POST /runs/{run_id}/follow-up

分别映射 Pi SDK：

session.steer(text)
session.followUp(text)

用户可以在 Agent 运行过程中追加：

停止修改代码，只做分析
重点检查数据库部分
忽略当前测试，先修复配置
执行完成后再生成一份报告

指令必须绑定到正确的 Run 和 Agent Session，不得跨 Conversation 串线。

⸻

4.8 支持可恢复审批

审批不能继续采用 Tool 内部固定时间轮询。

需要实现：

running
→ waiting_approval
→ 保存 Agent Checkpoint
→ 释放 Agent 执行资源
→ 用户批准或拒绝
→ 恢复 Agent Session
→ running / rejected

审批等待时间不应受单次 HTTP 请求或 Worker 生命周期限制。

Agent、API 或 Runner 重启后，审批仍然可以继续处理。

⸻

4.9 增加 Agent Run Budget

每个 Run 应支持以下限制：

max_steps
max_tool_calls
max_run_duration
max_llm_tokens
max_cost
max_consecutive_tool_failures
max_processes

接近预算时，Agent 应收到明确提示并开始收敛结果。

超过预算后，应产生明确终态：

budget_exceeded

不得无限调用模型和工具。

⸻

4.10 模型能力由 Registry 驱动

不再通过代码硬编码所有模型能力。

Model Registry 至少保存：

provider
model_id
api_protocol
input_modalities
context_window
max_output_tokens
supports_tool_call
supports_developer_role
supports_reasoning
thinking_levels
pricing
enabled

创建 Agent Session 时，应根据 Registry 获取真实模型配置。

5. Non-Goals

本任务不负责以下内容：

1. 不负责动态创建或销毁 Docker/K8s Runner。
2. 不负责实现一次 Run 一个 Pod。
3. 不负责完整的 Workspace Mount Namespace 隔离。
4. 不负责替换现有用户认证体系。
5. 不负责重新设计前端整体 UI。
6. 不负责实现多 Agent 协同或 Agent-to-Agent 协议。
7. 不负责实现通用 Workflow/DAG 引擎。
8. 不负责重新实现 pi-coding-agent 内部 Agent Loop。
9. 不负责恢复已经废弃的 Python Agent Runtime。

当前正式 Agent Runtime 继续采用独立 Node.js Agent Service。

7.2 Session Restore Flow

        ↓
查询 conversation.agent_session_id
        ↓
查询完整 agent_session_entries
        ↓

        ↓

12. Acceptance Criteria

12.1 Agent Session

* 同一 Conversation 连续发送三轮消息，只对应一个逻辑 Pi SDK Session。
* 第二轮能够恢复第一轮 Tool Call 和 Tool Result。
* Agent 服务重启后，可以继续原 Conversation。
* Compaction 后重启，Session 仍可恢复。
* Session Restore 失败时，不允许静默创建新 Session。
* 不再只依赖最后 40 条纯文本消息恢复上下文。

12.2 Process Execution

* Agent 能启动一个持续运行的 Web Server。
* Agent 能读取该进程实时日志。
* Agent 能向进程发送输入。
* Agent 能停止该进程。
* Agent Service 重启后，Process 状态可被识别。
* Run 取消后，关联进程必须真正停止。

12.3 Tool Execution

* 每个 Tool Call 都有唯一 tool_call_id。
* 每个 Tool Call 都产生完整 Ledger。
* Tool HTTP 响应丢失后，不会重复执行副作用。
* Tool Result 能够在 Session 恢复后重新使用。
* edit 多次匹配时必须拒绝静默修改。
* 文件修改必须返回 Diff。

12.4 Attachment

* 用户上传多个文件时，Agent 能准确识别本轮附件。
* Agent 不需要扫描全部 Uploads 目录猜测文件。
* 附件路径、文件名和 MIME Type 可被完整审计。

12.5 MCP

* Agent 能发现已注册 MCP Server 的 Tools。
* 未授权用户无法调用受限 MCP Tool。
* 高风险 MCP Tool 必须进入 Approval。
* MCP Tool Call 完整进入 Tool Ledger。

12.6 Runtime Interaction

* 用户可以在 Run 执行期间发送 Steer 指令。
* 用户可以提交 Follow-up 指令。
* Steer 不得发送到其他 Conversation。
* Agent 能在等待审批时释放执行资源。
* Agent 重启后能继续等待或恢复审批。
* 超过 Run Budget 后任务正确终止。

12.7 Model Registry

* 不同模型使用各自真实 Context Window。
* 不同模型使用各自真实输出限制。
* Model Registry 能标识 Tool Calling 和 Reasoning 能力。
* Run 记录实际模型、Token 和成本。

14. Risks

14.1 SDK Session 格式兼容

Pi SDK Session JSONL 格式可能随版本变化。

应保存：

sdk_version
session_schema_version

并实现迁移策略。

14.2 Session 数据规模增长

完整保存 Tool Result 和日志会导致数据库快速增长。

需要：

* 大日志外置存储。
* Tool Result 摘要。
* Compaction。
* Retention Policy。
* Artifact 化大结果。

14.3 长进程资源占用

后台进程可能长期占用 Runner。

需要：

* Process TTL。
* Runner 并发限制。
* 用户配额。
* 强制回收策略。
* 资源监控。

14.4 恢复后的副作用重复

Tool 执行结果不确定时，恢复 Run 可能重复调用 Tool。

必须依赖：

Tool Ledger
Idempotency Key
Unknown State
人工确认机制

14.5 范围过大

该任务覆盖多个 Runtime 核心能力，应按 Phase 实现，但所有 Phase 都归属于同一总体目标。

P0 最低交付范围为：

Agent Session 完整持久化恢复
Process Manager
Tool Ledger 全覆盖
日志流式输出

其余能力可在后续 Phase 完成。

⸻

15. Definition of Done

只有同时满足以下条件，本任务才允许归档：

1. 不再默认每轮创建独立的空 Pi SDK Session。
2. 同一 Conversation 可以恢复完整 Agent Session。
3. Tool Call 和 Tool Result 可以跨重启恢复。
4. Agent 能执行和管理长时间进程。
5. 命令输出支持实时流式读取。
6. 所有 Tool Path 接入统一 Ledger。
7. 附件以结构化方式传给 Agent。
8. Agent 能调用注册的 MCP Tools。
9. 支持 Steer 和 Follow-up。
10. 审批支持持久化暂停和恢复。
11. Run Budget 生效。
12. Model Registry 替代硬编码模型能力。
13. 所有验收测试通过。
14. 不存在 Deferred、Partial 或仅文档声明完成的核心项。
15. 若仍有残留，必须创建新的 Active Task，不得只写入归档备注。
