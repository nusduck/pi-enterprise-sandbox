3. Refactor Goal

前端需要从：

Chat UI

重构为：

Agent Runtime Workbench

新的前端需要同时表达三个层次：

Conversation
    └── Agent Session
            └── Agent Run
                    ├── Message
                    ├── Tool Execution
                    ├── Process
                    ├── Approval
                    ├── Artifact
                    └── Runtime Event

用户既能进行自然语言对话，也能明确了解：

* Agent 当前正在做什么
* 执行到了哪个步骤
* 哪些工具已经执行
* 哪些进程仍在运行
* 是否正在等待审批
* 哪些文件被修改
* 当前使用哪个模型
* Run 还剩多少预算
* 中断后是否可以恢复

4. Design Principles

4.1 Conversation 与 Run 分离

Conversation 是长期交互空间。

Run 是一次具体执行过程。

前端不能再只使用：

isStreaming = true / false

描述所有运行状态，而需要明确支持：

queued
restoring_session
running
waiting_approval
waiting_input
cancel_requested
cancelled
succeeded
failed
interrupted
budget_exceeded
orphaned

⸻

4.2 Chat 与 Runtime Timeline 分离

Assistant 的自然语言回复属于 Chat。

Tool、Process、Approval、Artifact 和系统状态属于 Runtime Timeline。

不能继续把所有内容都塞入一个 Assistant Message Bubble。

⸻

4.3 权威状态来自服务端

以下状态必须以服务端为准：

Agent Session
Agent Run
Tool Execution
Process
Approval
Artifact
Budget
Model Usage

LocalStorage 只保存：

* UI 偏好
* Sidebar 宽度
* 最近访问的 Conversation
* 展开和折叠状态
* 草稿文本

不再用 LocalStorage 作为 Conversation 消息的恢复来源。

⸻

4.4 支持断线和恢复

页面刷新、网络中断或切换 Conversation 后，前端应能够：

根据 run_id 查询当前状态
根据 Last-Event-ID 恢复 SSE
重新加载未完成 Tool
重新加载等待中的 Approval
重新连接 Process Log
继续显示已有 Artifact

7. Composer Refactor

Composer 根据 Run 状态切换模式。

7.1 Idle Mode

输入新任务
上传附件
选择模型
选择 Skill
发送

7.2 Running Mode

当 Agent 正在运行时，Composer 不应完全禁用。

提供：

Steer
Follow-up
Stop

Steer 示例：

停止修改代码，只分析问题。

Follow-up 示例：

当前任务完成后再生成测试报告。

界面需明确两者区别：

* Steer：尽快改变当前执行方向
* Follow-up：当前 Run 完成后继续执行

7.3 Waiting Approval Mode

Composer仍可用，但界面顶部显示：

用户可以：

* 审批
* 补充说明
* 取消 Run

14. SSE Architecture

当前单个 /chat 请求同时承担创建 Run 和消费 SSE。新的前端应拆分为：

POST /runs
→ 返回 run_id
GET /runs/{run_id}/events
→ SSE

SSE Manager 按 run_id 管理连接：

runId
lastEventId
connectionStatus
retryCount
abortController

必须支持：

* Event Sequence
* Last-Event-ID
* 自动重连
* 事件去重
* 乱序检测
* Run 完成后关闭
* Conversation 切换后保持后台运行
* 多个 Run 状态同时更新

页面切换不应默认取消 Run。

只有用户点击 Stop，才请求后端取消。

20. Acceptance Criteria

Core Compatibility

* 原有 Conversation 功能保持可用。
* 原有 Message Streaming 保持可用。
* 原有附件上传、重试和删除保持可用。
* 原有 Approval 和 Artifact 功能保持可用。

Session and Run

* 页面能区分 Conversation、Agent Session 和 Agent Run。
* 刷新页面后能恢复正在运行的 Run。
* 切换 Conversation 不会自动取消后台 Run。
* 同时存在多个 Run 时能分别更新状态。
* Interrupted Run 可以显示恢复入口。

Runtime Timeline

* Tool 不再仅作为普通 Message 内容展示。
* Tool Arguments、Result、状态和耗时可查看。
* Process Log 能实时显示。
* Approval 能跨页面持续显示。
* Artifact 能追溯到对应 Run 和 Tool。

Composer

* Idle 时可发送新任务。
* Running 时可发送 Steer。
* Running 时可添加 Follow-up。
* 用户可主动停止 Run。
* 上传附件能明确绑定本轮 Message。

Recovery

* SSE 断开后自动重连。
* 重连后不会重复显示事件。
* 使用 Sequence 恢复遗漏事件。
* 页面刷新后能重新加载未完成 Approval 和 Process。
* 错误提示包含 Trace ID 和推荐操作。

Engineering

* 新增代码使用 TypeScript。
* API 和 SSE 数据经过 Schema 校验。
* Server State 与 UI State 分离。
* 单个页面文件不得承担全部业务编排。
* 核心状态流具有单元测试。
* 关键用户流程具有 E2E 测试。

⸻

21. Non-Goals

本任务不负责：

* 重新设计 Agent 后端执行逻辑。
* 实现 Process Manager 后端。
* 实现 Pi SDK Session 恢复后端。
* 实现 MCP Server。
* 实现用户和组织权限后端。
* 实现动态 Runner 管理。
* 实现完整 IDE 或 VS Code 替代品。
* 实现多人实时协作编辑。

前端只消费对应后端能力，并提供一致的交互体验。

⸻

22. Definition of Done

满足以下条件后才允许归档：

1. 前端从 Chat 页面升级为 Agent Runtime Workbench。
2. Conversation、Session、Run 三层模型在 UI 和代码中明确分离。
3. Tool、Process、Approval 和 Artifact 使用独立实体。
4. 页面刷新和 SSE 重连后能恢复 Run。
5. 支持 Steer、Follow-up、Cancel 和 Resume。
6. 支持 Process 实时日志和管理。
7. 支持持久 Approval Center。
8. 支持 Run Budget 和 Model 信息展示。
9. API 和 Event Schema 完成类型化。
10. 旧的单文件业务编排完成拆分。
11. 原有功能没有回归。
12. 所有关键验收和 E2E 测试通过。
